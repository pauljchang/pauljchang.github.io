AI Engineering — Lesson 1

Building a Web App with AI

A hands-on walkthrough of the real AI-assisted development workflow — iterative, critical, and always engineer-led.

Duration ~90 minutes
What you'll build Expense Tracker with live chart
Stack HTML · JavaScript · localStorage · Chart.js
AI tool Claude, ChatGPT, or Gemini
Prerequisites A browser. Nothing else.

Learning Objectives

  1. Explain why AI-assisted development is iterative, not single-shot
  2. Write a clear, scoped prompt that produces useful code
  3. Read and critically evaluate AI-generated code before running it
  4. Recognize what HTML, CSS, JavaScript, and localStorage each do
  5. Use follow-up prompts to extend a working app without breaking it
Part 0

What AI Coding Actually Looks Like

10 min

Before writing a single prompt, let's set expectations.

Most people imagine using AI to write code goes like this: describe the app → AI builds it → done. That's not what happens. The reality is closer to pair programming with a very fast, sometimes overconfident collaborator. You drive. It codes. You review everything. You catch its mistakes. You decide what to ask for next.

The skill being developed today isn't prompting tricks. It's engineering judgment applied to an AI workflow — knowing what to ask for, reading what comes back critically, and knowing when something is wrong even before you run it. That judgment is what separates someone who uses AI well from someone who just runs whatever it produces.

The stack — all in one file

The app we're building lives entirely in a single file called index.html. Open it in a browser and it runs. No installation, no server, no terminal.

index.html
├── HTML         — the structure of the page (form, table, headings)
├── CSS          — how it looks (colors, layout, spacing)
└── JavaScript   — what it does (respond to clicks, do math, update the page)
         │
         └── localStorage   — the browser's built-in storage
                              survives page refresh; lives on this computer

You don't need to fully understand this stack before we start. You'll see it emerge from the AI output, and we'll explain each piece as it appears.

How to run it

  1. Save the AI's output as a file named index.html on your Desktop
  2. Double-click the file — it opens in your browser
  3. After any change, save the file and refresh the browser tab (Ctrl+R / Cmd+R)

That's the entire development loop for today.

Part 1

Design Before You Prompt

15 min

The single biggest mistake beginners make with AI coding tools is jumping straight to a prompt without knowing what they want. The AI will fill every gap you leave — sometimes sensibly, often not. Your spec is your steering wheel.

Exercise — Write the spec first

Before touching any AI tool, answer these four questions in writing:

  1. What can a user do with this app? List every action.
  2. What information does the app need to store for each expense?
  3. What should the page always show the user?
  4. What should happen if a user submits the form with a missing or invalid field?

Take 8 minutes. Write it down before reading ahead.

Compare your answers to the spec below. These should be close — and if they're not, that difference is worth discussing.

Agreed Spec

A user can add an expense (amount, category, description, date) and delete any existing expense. Each expense is saved to localStorage so data survives a page refresh. The page always shows: a form to add expenses, a table of all expenses, a summary of total spending per category, and a pie chart of that breakdown. Empty fields and non-positive amounts must be caught before saving.

This spec is your first prompt. You wrote it in plain English before touching the AI. That order matters — the thinking was yours, not the AI's.
Part 2

Iteration 1 — Scaffold the App

20 min

What makes a good first prompt

Compare these two prompts for the same task:

Weak: "Build me an expense tracker in HTML."
No stack specified, no data model, no behaviors defined. The AI will make every decision. You'll spend more time untangling its assumptions than you saved.

Strong: everything below. Notice that it specifies the technology, names each field, describes each behavior, and asks for a single complete file.

Use this prompt exactly as written. You're welcome to read it first and modify it — but try it as-is at least once so you can compare.

Prompt 1 — Paste into Claude / ChatGPT / Gemini
Build a single-file expense tracker web app using only HTML, CSS, and JavaScript — no frameworks, no libraries except Chart.js, no backend. Everything goes in one file called index.html. The app should: - Have a form at the top with four fields: Amount (number input), Category (text input), Description (text input), and Date (date input that defaults to today) - Validate on submit: Amount must be a positive number, Category must not be empty. Show a clear error message if validation fails — do not save the expense. - Save each valid expense to localStorage as a JSON array so data survives page refresh - Display all saved expenses in a table below the form with columns: Date, Category, Description, Amount, and a Delete button per row - Compute and display a summary table showing total spending per category, sorted highest to lowest - Load Chart.js from the jsDelivr CDN and render a pie chart of spending by category, placed below the summary table - Normalize all category names to Title Case before saving (so "food" and "Food" both become "Food") - When there are no expenses yet, show a friendly message in place of the table and chart Use clean, readable CSS. Generate the complete index.html file and nothing else.
If the AI asks clarifying questions instead of generating code, reply: "Go ahead and make reasonable choices — generate the full file."

Save and open the file

  1. Copy the AI's entire output
  2. Open a plain text editor (Notepad on Windows, TextEdit on Mac in plain-text mode)
  3. Paste and save as index.html
  4. Double-click the file to open it in your browser
Screenshot
The app running in a browser after Prompt 1 — form, empty state message visible

Read the code before you do anything else

Do not add an expense yet. Read the code first.

Skim through the file and find answers to these questions:

  • Where does the app load expenses from localStorage on startup?
  • What happens step-by-step when the Add button is clicked?
  • Where does the category summary get calculated?
  • Where does Chart.js get told what to draw?
  • What does the AI do when there are no expenses — is it what you expected?
We haven't written a line of code, but we just read all of it. That's not optional — that's the job. If you can't explain what this code does, you don't own it yet, and you can't responsibly build on top of it.

Now test it

Add a few expenses across two or three categories. Confirm:

  • Expenses appear in the table immediately
  • The category summary updates
  • The pie chart renders
  • Refreshing the page keeps your data
  • The Delete button removes a row

Write down anything that looks wrong or feels missing. You'll need that list in Part 3.

Part 3

Iteration 2 — Fix and Tighten

15 min

Something in the app is probably slightly off. That is not a failure — it's the lesson. The first output is never the final output.

Common things AI gets wrong on the first pass:

  • Deleting an expense doesn't update the chart
  • Category totals display too many decimal places (e.g. $12.300000000001)
  • The pie chart crashes or looks odd when only one category exists
  • The empty state message doesn't disappear after the first expense is added
  • Error messages don't clear after a successful submission

Look at your list from Part 2 and pick the most important issue. Then use the prompt below, editing the bracketed section to describe your specific problem.

Prompt 2 — Follow-up fix (edit before sending)
In the expense tracker we just built, I found this issue: [DESCRIBE THE PROBLEM HERE — be specific about what you did and what happened] Please fix it. Show me only the JavaScript functions that need to change, not the entire file. Explain briefly why the bug occurred.
Replace the bracketed section with your actual problem before sending. The more specific you are, the better the fix will be.

When you receive the fix, read it before applying it. Do you understand what changed and why? If not, ask the AI to explain it.

To apply a partial fix: find the function in your index.html, select it, and replace it with the AI's updated version. Save and refresh.

Two things to watch out for

Ask for diffs, not rewrites. Once you have a working base, asking for targeted changes is safer than regenerating the whole file. Regenerating risks losing your edits and introducing new problems.

The AI can drift. In a long conversation, the AI may subtly forget earlier constraints. This is why you read every output, not just the first one.

A good follow-up prompt is specific about what broke, references "the existing app," and asks for the minimum change needed — not a full rewrite.

Part 4

Iteration 3 — Add a Feature

15 min

The app works correctly now. This iteration adds one new feature: a running monthly total displayed prominently above the expense table.

Notice how this prompt is scoped differently from Prompt 1. It's smaller and more specific because we have more to protect. As an app matures, prompts should get narrower — not broader.

Prompt 3 — Add a monthly total
Add a "This Month" summary to the existing expense tracker. It should: - Display the total amount spent in the current calendar month - Appear between the form and the expense table - Update automatically when expenses are added or deleted - Show $0.00 when there are no expenses this month Show me only the new HTML to insert and the JavaScript changes needed. Do not regenerate the whole file.
After applying this change, add a few expenses dated in prior months and confirm they're excluded from the monthly total.
Screenshot
The monthly total displayed between the form and the expense table
We added a real feature in one focused prompt. But we decided what the feature was, what its edge case was (prior months), and how to test it. The AI executed our decision — it didn't make it.
Part 5

Reflection

10 min

Discuss these questions:

  1. What decisions did the AI make that you never explicitly asked for? Were they good decisions?
  2. At what point in the lesson did you feel most like an engineer? When did you feel like a typist?
  3. What would have happened if you'd submitted the first output without reading it?
  4. Right now, your data lives only in your browser. What breaks when two people need to share the same expense list?

That last question doesn't have an answer today. It's the question that motivates Lesson 2.

Five Takeaways

  1. Design first. A good prompt starts with a spec. The spec comes from thinking, not from the AI.
  2. Iteration is the workflow. One shot never gets you there. Budget for three to five rounds.
  3. Read everything. You are responsible for the code. The AI doesn't have bugs — you do.
  4. Scope your follow-up prompts. "Fix this one thing" beats "rewrite it all." Smaller prompts mean more control.
  5. AI raises your floor. What once took a day now takes an hour. But your ceiling is still your engineering judgment.
Take-Home

Extend the App on Your Own

Choose one of the following features and add it using the AI. You must write a one-paragraph spec before writing each prompt. Both the spec and the prompt must be included in your submission.

  • Monthly filter — a dropdown that shows only expenses from a selected month
  • Spending limit — set a per-category budget; show a warning when a category exceeds it
  • Edit button — click to modify a saved expense in place
  • CSV export — a button that downloads all expenses as a .csv file
