Why does word order matter?

Lesson 1's model predicted each word from the average of every embedding before it. Averaging has a hidden problem: mean(a, b) is mathematically identical to mean(b, a) — addition doesn't care what order you add things in. That means, as far as this model is concerned, "hunter likes" and "likes hunter" look exactly the same.

The obvious fix — give every position its own extra vector and add it in before averaging — doesn't actually work, and it's worth seeing why: mean(A+P0, B+P1) still equals (A+P0+B+P1)/2, which is the same four numbers added together no matter which token got paired with which position. Averaging erases that pairing regardless of what you add beforehand. The real fix has to stop averaging: instead of blending every position into one vector, we concatenate them into fixed slots — position 0's contribution and position 1's contribution stay in separate places, so the model can finally tell which word came first. Real transformers avoid this whole problem differently: they still add position vectors to token vectors, but instead of averaging afterward, they feed everything into attention — which combines positions with content-dependent weights instead of a flat average. Fixed-slot concatenation is the version of "don't average" simple enough to build before Lesson 3 introduces that properly. Why not just always concatenate, forever? ▾ Because concatenation needs a fixed maximum length — every extra position means a bigger weight matrix, and sentences longer than that maximum simply don't fit. That's fine for a toy world capped at 3-word contexts, but it doesn't scale to real sentences of wildly varying length. Attention solves both problems at once: it uses position information without pooling, and without needing a fixed-size slot for every possible position.

With the architecture OFF (Lesson 1's pooling), the two representations — and the model's predictions — are identical for every pair here, down to floating-point noise, because averaging can't see order no matter how long you train. With it ON, the fingerprints are always different once concatenation preserves each slot. For the real grammatical pair (hunter likes / dog likes), the predictions become meaningfully different too — correctly landing on dog and hunter respectively, matching what's actually true in the corpus. For the reversed nonsense pair (likes hunter), the representation is still different, but the prediction isn't "correct" in any meaningful sense — the model was never trained on that word order, so it's just extrapolating, the same way it would for any other input it's never seen.
