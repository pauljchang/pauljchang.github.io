The Apollo Problem

In 1966, North American Rockwell faced a data problem unlike any the world had seen. They had been contracted to build the Saturn V — the rocket that would carry humans to the Moon — and it had a bill of materials comprising approximately three million individual parts. Each part had a part number, a specification, a supplier, a delivery schedule, and a precise location within the vehicle. Many had sub-components. Sub-components had their own sub-components. The whole thing was a tree of physical containment, and it went six or seven levels deep.

At the time, there was no good way to store this structure in a computer. Files were flat. You could store a list of parts, but the relationships between them — which engine assembly a particular turbopump belonged to, which stage that engine belonged to — were implicit, hand-maintained, and fragile. Rockwell's engineers were tracking dependencies in paper ledgers. For a mission where a single misplaced decimal could kill three astronauts, this was unacceptable.

Rockwell brought in IBM. Working also with Caterpillar Tractor, who had similar hierarchical inventory problems, IBM's team spent two years designing a solution. They called it IMS — Information Management System. It shipped in 1968. Twelve months later, Neil Armstrong walked on the Moon.

"The idea was simple: if the physical structure of the rocket is a tree, store it as a tree. Navigate it the same way an engineer would — start at the vehicle, descend to the stage, descend to the engine, descend to the assembly, find your part."

— IBM IMS Design Philosophy, 1967

IMS organized data into segments arranged in a strict parent-child hierarchy. The query language, called DL/I (Data Language One), worked by specifying a path through the tree. To find the oxidizer turbopump, you told DL/I: get the ROCKET segment named "Saturn V", then get its STAGE child named "S-IC", then get its ENGINE child where position = 1, then get the ASSEMBLY child named "Turbopump", then get its PART where name = "Oxidizer". The path was the query.

For its intended purpose, this was not just adequate — it was elegant. Physical containment is genuinely hierarchical. A turbopump does belong to exactly one engine. An engine belongs to exactly one stage. The tree structure matched reality, and IMS exploited this to store records in physical proximity on disk, making reads along known paths extraordinarily fast on the hardware of 1968.

IMS never went away. Today, some of the largest banks in the world — institutions processing tens of millions of transactions per day — still run IMS on IBM mainframes. Migration is simply too risky and too expensive. Estimates suggest IMS handles over 50 billion transactions per day globally. The database built for the Apollo program remains critical infrastructure for the global financial system. That is not a coincidence: both problems, at their core, involve hierarchical structures — accounts within branches within regions within banks — that IMS handles with extreme reliability and speed.

The wall appeared when data stopped being truly hierarchical. A single bolt — part number P/N-4521 — might be used in twelve different assemblies across three engine types. In a tree, a node can have only one parent. The bolt cannot belong to the thrust chamber assembly and the turbopump assembly and the gas generator assembly simultaneously. Rockwell's engineers had two choices: store twelve copies of the bolt record (risking inconsistency when the spec changed) or use IMS's "logical child" pointers — a mechanism that grafted a pointer from one tree onto another. The pointer hack worked, but it was inelegant, fragile, and invisible to the query language. IMS was smuggling relational structure in through the back door.

In 1970, a researcher named Edgar F. Codd — working at IBM's San Jose Research Laboratory, the same company selling IMS — published a paper titled "A Relational Model of Data for Large Shared Data Banks." It proposed organizing all data as flat tables linked by shared keys, with a mathematical foundation in set theory and relational algebra. Instead of navigating a tree, you described what you wanted and let the system figure out how to retrieve it. The paper made IMS's fundamental limitation explicit and proposed an architecture that would eventually displace it everywhere except where IMS was already too deeply embedded to remove.

Lab Exercises

EX 01
🌲 Finding the Wall
Use Demo 1 to identify exactly when and why the hierarchical model breaks down.

Work through all six scenarios in Demo 1 in order. For each "green" scenario, describe in one sentence why the data is naturally hierarchical (what makes one parent per node the right model). For each "red" scenario, describe the relationship that breaks the tree and what you were forced to do about it.

→ In the University Enrollment scenario, suppose you choose "Duplicate" for every conflict. If a student transfers to a new section, how many records need updating? What happens if one is missed?

Key question: The Saturn V bolt was a genuine problem for IMS. Why was it less catastrophic for a rocket BOM than it would be for a hospital patient record system? (Think about how often the data changes, and what the consequence of inconsistency is.)

EX 02
🔍 Query Asymmetry
Use Demo 2 to identify which query style suits each system better.

Run all five queries in Demo 2. For each one, decide: does IMS or SQL handle it more naturally? What is the structural reason?

→ Notice that Query 1 (path lookup) is almost easier to read in DL/I than SQL — the DL/I notation directly maps to how an engineer thinks about the problem ("go to Stage 1, Engine 2, Turbopump"). Under what circumstances might DL/I's navigational style be an advantage even today?

Key question: Query 3 asks "which assemblies use Bolt P/N-4521?" This is a bottom-up query in a top-down system. In the IMS schema, why is this query fundamentally harder than in SQL, and what does this reveal about the difference between navigational and declarative query models?

EX 03
🧠 Codd's Insight
Think through what made the relational model a genuine breakthrough rather than just a refinement.

In the relational schema, the asm_parts junction table stores one row per (assembly, part) combination. This seems simple — almost too simple. Explain why this one table eliminates all three problems that IMS faced with the bolt: duplication, inconsistency risk, and the difficulty of bottom-up queries.

→ Codd worked at IBM while IMS was IBM's flagship product. His 1970 paper implicitly argued that IMS was architecturally flawed. Why do you think IBM initially resisted commercializing the relational model? (Consider: IBM had thousands of IMS customers and a large consulting business built around it.)

Key question: IMS still runs billions of transactions per day in 2025. Does this mean the hierarchical model was "good enough," or does it mean something different — about the economics of replacing working infrastructure? What would it cost a large bank to migrate from IMS to a modern relational system?

05

Discussion Questions

  • The Right Tool. IMS was purpose-built for a specific problem — hierarchical inventory tracking — and it excelled at it. Modern developers often reach for general-purpose relational databases by default. Can you think of current problems where a hierarchical store (like a filesystem, an XML database, or a modern document store like MongoDB) might be the more natural choice than a relational database?
  • Path Dependence. The fact that IMS is still running in 2025 — nearly 60 years after its design — is an example of technological path dependence: early choices constrain future options even when better alternatives exist. What other examples of path dependence can you identify in computing? (Consider keyboard layouts, IP addressing, or file formats.)
  • The Codd Paradox. Codd published his relational model paper while working for IBM, whose main database product (IMS) it implicitly criticized. His paper was initially suppressed internally and only published after persistence. What does this suggest about how genuinely disruptive ideas travel through large organizations? How does this dynamic show up in modern tech companies?
  • From Trees to Graphs. A hierarchical database is a tree (each node has one parent). A relational database is, conceptually, a set of flat tables. Modern graph databases (like Neo4j) allow any node to connect to any other node. For what kinds of problems is a graph the most natural structure? Think about social networks, knowledge graphs, or route planning.
  • Connection Forward. In Era 2 you will study relational databases and SQL in depth. The junction table (asm_parts) that solves the bolt problem is the foundation of the entire relational model. What happens to this approach when a database has billions of rows and thousands of simultaneous queries? This is the scalability question that drove the NoSQL movement in the 2000s — which you'll reach in Era 3.
06

Terminology

IMSInformation Management System
IBM's hierarchical database system, developed 1966–68 for the Apollo/Saturn V program. Organizes data as a tree of segments (records), where each segment has exactly one parent. Still in production use at major banks and airlines. Query language: DL/I.
DL/IData Language One
IMS's navigational query language. Queries specify an explicit path through the tree hierarchy. Efficient for known top-down traversals; struggles with bottom-up or cross-branch queries. Contrast with SQL, which is declarative (you describe what you want, not how to get it).
Hierarchical DBTree-structured data store
A database where every record has exactly one parent (except the root). Naturally models containment hierarchies: org charts, file systems, bills of materials. Cannot represent many-to-many relationships without duplication or pointer hacks.
Logical ChildIMS pointer mechanism
IMS's workaround for many-to-many relationships. A segment can have a logical parent in a second tree, accessed via pointer. Functional but fragile: the relationship is invisible to standard queries and requires manual maintenance. Essentially smuggles relational structure into a hierarchical system.
Bill of MaterialsBOM
A hierarchical list of components, sub-components, and raw materials needed to manufacture a product. The Saturn V BOM had ~3 million entries across 6–7 levels. BOMs are one of the canonical cases where a hierarchical data model genuinely fits the problem.
Relational ModelCodd, 1970
Edgar Codd's proposal to organize all data as flat tables linked by shared keys, with queries expressed in relational algebra. Addresses the hierarchical model's many-to-many problem via junction tables. Foundation for SQL and every major database system used today.
Junction TableBridge / associative table
A table in a relational database that stores many-to-many relationships by holding foreign keys to both related tables. Example: asm_parts(assembly_id, part_number, quantity) connects assemblies to parts without duplication. Each part exists exactly once; each assembly-part relationship gets one row.
