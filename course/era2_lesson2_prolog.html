02

Code Is Data — and Logic Is Code

John McCarthy invented Lisp in 1958 — the same year Rosenblatt announced the perceptron. The timing was not coincidental. McCarthy was building the theoretical foundations of AI at MIT, and he needed a language that could manipulate symbols the same way mathematics manipulates symbols. Numbers were well handled by existing languages. What McCarthy needed was a language in which a list of words, a logical rule, and a program were all the same kind of object — something you could store, pass around, and transform.

He got this from a radical design decision: in Lisp, code and data share the same syntax. A function call looks like (classify robin). A list of facts looks like (has-feathers robin). Both are s-expressions — nested lists of atoms. This property, called homoiconicity, means that a Lisp program can manipulate another Lisp program as easily as it manipulates a list of numbers. An inference engine, written in Lisp, is just a function that takes a list of rules and a list of facts and returns a list of conclusions. The engine and the knowledge it reasons about are made of the same material.

For thirty years, this made Lisp the language of AI. DENDRAL was Lisp. MACSYMA — the first computer algebra system — was Lisp. Most of the expert systems built at Stanford and MIT were Lisp. The language also pioneered features that are now ubiquitous: garbage collection (McCarthy invented it for Lisp), dynamic typing, closures, and the REPL — a read-eval-print loop that let researchers interact with their programs in real time, in 1958, when everyone else was submitting punch cards.

"Lisp has jokingly been called 'the most intelligent way to misuse a computer.' I think that description is a great compliment because it transmits the full flavor of liberation: it has assisted a number of our most gifted fellow humans in thinking previously impossible thoughts."

— Edsger Dijkstra, 1972

In 1972, a French computer scientist named Alain Colmerauer, working in Marseille with Philippe Roussel, took a different approach. Instead of giving programmers a general-purpose language and letting them build inference engines, why not make the language itself an inference engine? Colmerauer's insight was that a subset of first-order logic — specifically Horn clauses — could be used directly as a programming language. The interpreter would be a theorem prover. Writing a program meant stating logical facts and rules; running the program meant posing a query and letting the resolver find what was true.

The language was called Prolog — from Programmation en Logique. Its key mechanism was unification: a generalization of pattern matching in which variables can be bound to values by finding consistent substitutions that make two terms identical. Unification is to Prolog what addition is to arithmetic — the primitive operation from which everything else is built.

Prolog and Lisp represent genuinely different philosophies about what a programming language should be. Lisp says: here is a powerful notation for computation; build whatever you need. Prolog says: computation is proof; tell me what is true and I will find it. Both had profound influence on the development of AI — and both eventually ran into the same walls that stopped expert systems: the brittleness of closed-world reasoning, the difficulty of scaling to real-world knowledge, and the inability to handle uncertainty.

🔷 Prolog — Logic Programming

Declarative: you state what is true. The engine figures out how to prove it. Variables range over the entire database — a query finds all solutions automatically.

Strength: recursive relations, search, constraint solving. Used in: computational linguistics, knowledge graphs, planning.

ancestor(X,Z) :- parent(X,Y), ancestor(Y,Z).

🔶 Lisp — Symbolic Computation

Symbolic-procedural: you say how to compute, but with symbols as first-class data. The inference engine is itself a Lisp program — code and knowledge are the same kind of object.

Strength: meta-programming, flexibility, expressiveness. Used in: AI research, theorem provers, Emacs, Clojure, Racket.

(defun classify (x) (cond ((bird? x) 'bird) ...))

Lab Exercises

EX 01
🌳 Extend the Family Tree
Add new facts and rules to the family tree knowledge base — and discover how Prolog reasoning extends automatically.

Load the Family Tree scenario. Add a fact: parent(jim, sue). Re-run ancestor(tom, X). How many solutions now? The rule didn't change — only a fact was added, and Prolog automatically extended the transitive closure.

→ Add a rule: cousin(X, Y) :- parent(P1, X), parent(P2, Y), sibling(P1, P2). What does cousin(ann, X) return? Trace the resolution steps to verify.

Key insight: In Prolog, adding a fact doesn't require updating any function — the existing rules automatically incorporate it. Compare this to the Lisp classifier: if you add a new animal to the facts list, you only get correct results if the classify function's conditions happen to match. Rules and data are more cleanly separated in Prolog.

EX 02
🔗 Unification by Hand
Work through unification manually before checking with Demo 2.

Before using Demo 2, predict the result of each unification: (a) foo with foo (b) X with tom (c) parent(X, bob) with parent(tom, Y) (d) f(X, X) with f(tom, bob) (e) ancestor(X, Z) with ancestor(tom, ann).

→ For (d): f(X, X) with f(tom, bob). X must be tom (from first arg) AND bob (from second arg). These contradict. Predict: FAILS. Verify with Demo 2.

Key question: In the Prolog resolver, every time a rule is tried, its variables are renamed with a unique suffix (X becomes X_42, Y becomes Y_42, etc.). Why is this necessary? What would go wrong without renaming — try to construct a specific query that would fail incorrectly without it.

EX 03
🔷🔶 The Existential Query
Use Demo 3 to understand exactly why Prolog can answer "who?" while Lisp can only answer "is this one?"

In the Prolog panel, run: flying_bird(X). You get all flying birds. Now run: bird(X), \+(flying_bird(X)). You get non-flying birds. Neither of these queries requires knowing the animals in advance — Prolog searches the entire database.

→ In the Lisp panel, try: (classify 'eagle), (classify 'penguin), (classify 'shark). Works for specific animals. Now try to answer "what are all flying birds?" in Lisp — you would need to write something like (filter flying-bird? (all-known-animals)). The list of all animals must exist somewhere explicitly.

Key insight: Prolog's unification lets a variable range over the database — X in flying_bird(X) iterates over all possible bindings automatically. In Lisp, iteration requires an explicit data structure to iterate over. This is what "declarative" really means: in Prolog, you don't say how to search — you say what you want, and the engine searches for you.

05

Discussion Questions

  • The Homoiconicity Advantage. Lisp's code-is-data property meant that an inference engine in Lisp could treat rules as data to be manipulated — modified, extended, even generated at runtime. Modern neural networks also blur the line between code and data: the weights are the "program." How does a neural network's forward pass resemble a Lisp interpreter evaluating a nested s-expression? Where does the analogy break down?
  • The Closed World in Prolog. In Demo 1, try querying a predicate that doesn't exist in the knowledge base — for example, likes(tom, X) in the Family Tree. Prolog returns false — not "unknown." This is the closed-world assumption at the language level. Is it ever appropriate to treat "not in the database" as "false" rather than "unknown"? Give an example where it's safe and one where it's dangerous.
  • Prolog's Influence on Modern AI. Unification — Prolog's core mechanism — reappears in surprising places. Pattern matching in Haskell and Rust, destructuring in Python and JavaScript, type inference in strongly-typed languages, and (at a deep level) the Attention mechanism in Transformers all involve finding consistent assignments of variables to values. Where else in your experience with technology have you encountered something that works like unification?
  • The Fifth Generation Project. In 1982, Japan announced a ten-year, $400M project to build "fifth generation computers" using Prolog as the operating language — the idea was that logic programming would be as fundamental as assembly language, enabling AI at scale. The project ended in 1992 with few of its goals met. Given what you now know about Prolog's strengths and limitations, what specific properties of Prolog made this ambition plausible — and what would have prevented it from succeeding even with unlimited funding?
  • Connection to Era 3. Both Prolog and Lisp assume complete, correct, and consistent knowledge. Real-world knowledge is incomplete (there are things we don't know), incorrect (some of what we "know" is wrong), and inconsistent (sources contradict each other). The statistical learning revolution in Era 3 was partly a response to this — probabilities can represent partial belief, tolerate noise, and handle contradiction. Can you think of a way to add probabilistic reasoning to Prolog? (Hint: look up "Probabilistic Logic Programming.")

Terminology

Horn ClauseProlog's logical unit
A logical formula with at most one positive literal. Facts are Horn clauses with no body: parent(tom, bob). Rules are Horn clauses with conditions: ancestor(X,Z) :- parent(X,Y), ancestor(Y,Z). The entire Prolog language is built on Horn clauses — they are complete enough for many practical problems and efficient enough for a theorem prover to handle in real time.
head :- body1, body2, ..., bodyN. % rule head. % fact (body is empty)
UnificationProlog's pattern matching
An operation that finds a substitution of variables for values that makes two terms identical. parent(X, bob) unifies with parent(tom, Y) via {X=tom, Y=bob}. Fails if no consistent substitution exists: f(X,X) cannot unify with f(tom,bob) because X cannot simultaneously be both tom and bob.
SLD ResolutionProlog's proof strategy
The algorithm Prolog uses to prove a query: pick the leftmost goal, find all clauses whose head unifies with it, substitute the clause body for the goal (with bindings applied), and recurse. When all goals are empty, a solution is found. Backtracking occurs when a branch fails — the resolver tries the next matching clause.
HomoiconicityCode = Data
The property of a language in which programs and data share the same representation. In Lisp, both a list of facts and a function definition are s-expressions — nested lists of atoms. This means a Lisp program can generate, analyze, and modify other Lisp programs as easily as it processes any other data. Foundational to meta-programming and the design of inference engines in Lisp.
S-ExpressionLisp's syntax
A parenthesized, prefix-notation term used in Lisp: (operator arg1 arg2 ...). Function calls, data structures, control flow, and definitions all use the same syntax. (+ 1 2) is a function call. (1 2 3) is a list. (defun f (x) (* x x)) is a function definition. The uniformity of s-expressions is what makes homoiconicity possible.
Negation as Failure\+ in Prolog
The closed-world assumption made explicit: \+(Goal) succeeds if and only if Goal cannot be proved from the current database. Not the same as logical negation — it is an operational statement about the limits of the knowledge base. \+(flies(penguin)) succeeds because there is no fact flies(penguin) in the database, not because we have proved penguins don't fly.
