The Paper That Disrupted IBM

In June 1970, a researcher named Edgar F. Codd published a paper in the Communications of the ACM titled "A Relational Model of Data for Large Shared Data Banks." Codd worked at IBM's San Jose Research Laboratory. IBM's flagship database product was IMS — the hierarchical system we examined in Lesson 1.7, the one built for the Apollo program. Codd's paper implicitly argued that IMS was architecturally wrong.

The paper proposed something radical in its simplicity: store all data as flat tables. No trees, no pointers, no navigational paths. Each table is a mathematical relation — a set of rows (tuples) over a fixed set of columns (attributes). Relationships between tables are expressed not through physical pointers but through shared values: a patient table has a patient ID; an appointments table references that same patient ID as a foreign key. The connection is logical, not physical. You can traverse it in any direction without pre-defining the path.

IBM's response was complicated. The company knew the relational model was superior — Codd's mathematics were impeccable. But they had thousands of IMS customers, a large consulting business built around IMS, and no financial incentive to make their own product obsolete. They began building a relational system internally — called System R — but kept it experimental while continuing to sell IMS commercially.

"I had a running war with the IMS people. They said the relational model was theoretically interesting but practically useless. I said IMS was practically deployed but theoretically bankrupt."

— Edgar F. Codd, interviewed 1993

The gap IBM left was filled by a startup. In 1977, Larry Ellison read a published paper about IBM's System R — including its query language, which IBM had named SQL. Ellison recognized that IBM had not yet shipped a commercial product. He founded Software Development Laboratories, hired two engineers, and built a relational database from the System R paper. They named it Oracle. Oracle shipped in 1979, two years before IBM's own relational product, DB2. IBM's published research had handed Ellison the blueprint for what would become one of the most valuable software companies in history.

By 1986, ANSI had standardized SQL and it was the universal language of data. IMS retreated to the mainframe environments where it was too embedded to remove. The hierarchical model didn't disappear — it still runs billions of transactions daily — but it stopped being the default. From 1986 onward, when an organization needed a database, they built a relational one.

The intellectual core of Codd's contribution was deceptively simple: every fact should be stored in exactly one place. If a fact appears in two places, those two copies can diverge. Once they diverge, which one is true? This principle — called normalization — is the foundation of relational database design. A fully normalized database has no redundancy: no fact is repeated. And because no fact is repeated, there is no inconsistency to resolve.

Lab Exercises

EX 01
⚠️ Hunt the Anomalies
Use Demo 1 to identify all three types of anomaly in the denormalized table — before any normalization steps.

In Demo 1, stay on the "Denormalized" step. Trigger the update anomaly — Dr. Reyes moves from Cardiology to Neurology. How many rows need updating? What happens if one is missed?

→ Now think through the insertion anomaly: can you add a new doctor to the system without also creating a fake appointment? Why or why not? What does this imply about the structure of the table?

→ Think through the deletion anomaly: if you delete the only appointment for a particular doctor, what else gets deleted along with it? Is this a problem?

Key question: All three anomalies stem from the same root cause — one fact (the doctor's department) is stored in multiple rows. Why does storing one fact in multiple places inevitably create these three failure modes? Write a one-sentence explanation for each.

EX 02
📝 Write Your Own JOIN
Use Demo 2 to explore how JOIN connects tables that IMS could only connect with pointer hacks.

Run Query 3 (appointments with doctor names). Study the JOIN condition: ON a.doctor_id = d.id. This is a logical connection through a shared value — no physical pointer, no pre-defined traversal path.

→ Modify the query to also show the patient's name (you'll need a second JOIN to the patients table). What does the ON condition look like for the second join?

→ Run Query 5 (patients per doctor). Change COUNT(*) to COUNT(DISTINCT patient_id). Does the result change? Why might a doctor have multiple appointments with the same patient, and why does DISTINCT matter here?

Key question: In IMS (Lesson 1.7), finding all appointments for a given doctor required navigating a specific path through the tree. In SQL, you write WHERE doctor_id = 3 regardless of how the data is physically stored. What does this tell you about the difference between navigational (IMS) and declarative (SQL) query models?

EX 03
🔧 Design a Schema
Apply normalization principles to a new domain — without any scaffolding.

A school wants to track: students, the courses they take, the teachers who teach those courses, and the grades students receive. Start with a single flat table containing all this information. Write out what columns it would have.

→ Identify at least two update anomalies, one insertion anomaly, and one deletion anomaly in your flat table. Then decompose it into a normalized schema. How many tables do you end up with? What are the foreign key relationships?

→ In Demo 3, look at how the hospital schema handles the many-to-many relationship between patients and doctors (via the appointments table). Your student-course schema has the same many-to-many structure. What is the equivalent junction table? What columns does it need beyond the two foreign keys?

Key insight: A many-to-many relationship always requires a junction table in a relational database. This is the same insight that solved the bolt problem in the IMS lesson — asm_parts(assembly_id, part_number, quantity) was a junction table. The pattern is universal.

Discussion Questions

  • The Oracle Paradox. IBM published the System R paper — including the SQL query language — as academic research. Larry Ellison used that paper to build Oracle and ship it before IBM did. IBM's own research became the foundation of its primary competitor. What does this suggest about the relationship between open academic publication and commercial competition in technology? Should IBM have kept System R secret?
  • Normalization vs. Performance. A fully normalized database minimizes redundancy but can require many JOINs to answer a single question. In high-traffic systems (social media, financial trading, game servers), JOINs across large tables are expensive. Modern data warehouses often deliberately denormalize data — reintroducing redundancy — to improve query speed. Given what you know about the anomalies normalization prevents, what are the risks of intentional denormalization? What safeguards would you need?
  • SQL at 50. SQL was standardized in 1986 and is still the dominant data query language in 2025. Python is from 1991 and has largely displaced languages from the same era. JavaScript is from 1995 and has evolved dramatically. Why has SQL remained stable and dominant while most programming languages from the same period have been replaced or transformed? What does this suggest about the relationship between a language's design and its longevity?
  • Connection to Data Cleaning. In Lesson 3 you cleaned a patient dataset with NULLs, duplicates, and format inconsistencies. A properly normalized schema prevents many of these problems from arising in the first place. Which specific data quality issues from Lesson 3 would a normalized schema have prevented? Which would it not prevent? What does this suggest about the relationship between schema design and data quality?
  • Connection to Era 3 — NoSQL. In the 2000s, companies like Google, Amazon, and Facebook found that relational databases didn't scale to billions of users and petabytes of data. They built alternative systems — NoSQL databases — that sacrificed some relational guarantees for horizontal scalability. In Era 3, Lesson 4, you will study document stores like MongoDB. Based on what you've learned about the relational model, what specific relational properties do you predict NoSQL databases had to abandon? What did they gain in return?

Terminology

RelationMathematical table
Codd's term for a table: a set of tuples (rows) over a fixed set of attributes (columns). Each row is unique; order doesn't matter. A relation has no duplicate rows and no notion of "first" or "last." SQL tables are an implementation of relations, with some relaxations (SQL allows duplicate rows unless you use DISTINCT or a primary key).
Foreign KeyLogical pointer
A column in one table whose values reference the primary key of another table. The relational model's replacement for physical pointers. appointments.doctor_id references doctors.id. The connection is logical — enforced by the database engine — and traversable in any direction, unlike IMS's physical tree pointers.
appointments.doctor_id → doctors.id (logical, bidirectional) IMS parent segment → child segment (physical, top-down only)
NormalizationOne fact, one place
The process of organizing a database schema to minimize redundancy and eliminate update anomalies. Proceeds through normal forms: 1NF (atomic values, no repeating groups), 2NF (no partial key dependencies), 3NF (no transitive dependencies). The practical goal: every fact appears in exactly one place, so changing it requires updating exactly one row.
Update AnomalyThe redundancy tax
When updating a fact requires changing multiple rows, and missing one leaves the database in an inconsistent state. Example: a doctor's department stored on every appointment row — moving the doctor to a new department requires updating every appointment. Missing one row produces two different "true" departments simultaneously. Normalization eliminates this by storing the department in exactly one place.
JOINSQL's relational connector
A SQL operation that combines rows from two tables based on a matching condition (usually a foreign key relationship). INNER JOIN returns only rows with matches in both tables. LEFT JOIN returns all rows from the left table plus any matches from the right (unmatched right-side columns are NULL). JOINs are the SQL equivalent of IMS path traversal — but declarative and bidirectional.
SELECT p.name, d.name FROM appointments a JOIN patients p ON a.patient_id = p.id JOIN doctors d ON a.doctor_id = d.id
GROUP BYAggregation
A SQL clause that partitions rows into groups and applies aggregate functions (COUNT, AVG, SUM, MIN, MAX) to each group. The operation IMS had no native support for — requiring application-level loops. GROUP BY doctor_id with COUNT(*) counts appointments per doctor in a single declarative statement.
SELECT doctor_id, COUNT(*) AS n_appointments FROM appointments GROUP BY doctor_id ORDER BY n_appointments DESC
