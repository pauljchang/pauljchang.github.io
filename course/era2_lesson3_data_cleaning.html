02

The Warehouse That Wasn't

By the mid-1980s, large organizations had accumulated years of data in relational databases. Insurance companies had policy records. Hospitals had patient records. Banks had transaction records. Each department had built its own system, in its own format, with its own conventions. When the data warehouse movement arrived — the idea that you could consolidate all of an organization's data into a single analytical store — the first discovery was almost universal: the data was not what anyone thought it was.

A hospital attempting to consolidate patient records from two acquired clinics in 1987 found: dates of birth entered in six different formats. Patient names in all-caps in one system, title case in another. A blood type field that used "O+", "O positive", "O pos", and "0+" (with a zero) interchangeably. NULL values in the allergy field that meant three different things depending on which clinic entered the record — not collected, no known allergies, or the field didn't exist in the original paper form. And duplicate patients: the same person admitted under slightly different name spellings at different facilities, now appearing as two separate records with two separate medication histories.

This was not a rare situation. It was the standard situation. The lesson the data warehouse movement learned — slowly and expensively — was that data quality is not a technical problem, it is an organizational problem. The technical tools for cleaning data are not complicated. What is complicated is understanding the domain well enough to know what "correct" looks like, tracing the provenance of errors back to their source, and building processes that prevent the same errors from recurring.

"Data scientists spend 60% of their time on cleaning and organizing data. This is the least enjoyable part of the job — and, arguably, the most important."

— CrowdFlower Data Science Report, 2016

The phrase "garbage in, garbage out" predates computers — it was used by Charles Babbage in the 19th century to make exactly this point about mechanical calculation. But it took the database era to give the problem its modern form. A rule-based expert system like MYCIN could only fire rules on facts that were present and correct. A statistical model trained on dirty data learns the noise as signal. A query that silently ignores NULL rows returns a result that looks correct but isn't. Every downstream operation inherits whatever problems exist in the data, and most operations do not warn you that they're doing so.

2a

NULL: The Three-Headed Problem

When Edgar Codd introduced NULL in the relational model, he was making a philosophical commitment: a database should be able to represent the state of not knowing something, distinct from knowing it is false. The alternative — inventing sentinel values like 99999 for unknown age or "N/A" for unknown name — pollutes the data with fake values that then corrupt every calculation.

The problem is that a single NULL marker is being asked to represent three completely different situations, each requiring a different response:

Not Collected

The field existed but was never asked. Pre-2010 patient records have no email address — not because the patient doesn't have one, but because it wasn't collected.

email = NULL (record from 1987)

Strategy: usually leave as NULL. Filling in a value would be fabricating data.

Not Applicable

The field genuinely doesn't apply to this row. A male patient doesn't have a pregnancy date. A deceased patient doesn't have a next appointment.

pregnancy_date = NULL (male patient)

Strategy: often correct to leave as NULL, but may indicate a schema design problem.

Unknown

The field should have a value but we don't know it. The patient was unconscious on admission. The blood type wasn't on file. The allergy history was verbal and not recorded.

blood_type = NULL (unconscious admission)

Strategy: the most dangerous case. May require imputation, flagging, or active collection.

2b

Ternary Logic: When NULL Meets Boolean

Standard Boolean logic has two values: TRUE and FALSE. The moment you introduce NULL as "unknown," you need a third. SQL operates under three-valued logic — every logical expression can evaluate to TRUE, FALSE, or NULL (unknown). The rules are mostly intuitive, but a few cases are deeply counterintuitive and cause real bugs in production systems:

The highlighted rows are the ones that surprise people. NULL AND FALSE = FALSE — even though one operand is unknown, the other being FALSE guarantees the result is FALSE regardless of what the unknown turns out to be. Similarly, NULL OR TRUE = TRUE — the TRUE dominates. But NULL AND TRUE = NULL and NULL OR FALSE = NULL — when the definite value can't override the unknown, the result stays unknown.

The practical consequence is the most important: WHERE clauses only pass rows where the condition is TRUE. A condition that evaluates to NULL is treated the same as FALSE — the row is silently excluded. This means the query below does not return patients whose age is unknown:

⚠ The Silent NULL Trap — Interactive

The query returns 12 rows but gives no indication that 3 rows were dropped. A researcher counting "patients who are not 30" would report 12, missing the 3 unknowns. The correct query: WHERE age != 30 OR age IS NULL

Lab Exercises

EX 01
⚠ The NULL Trap
Predict what the following SQL queries return from a patient table where 15% of age values are NULL. No database required — use the ternary logic table.

For each query, state: (a) which patients are included, (b) which are excluded, (c) whether any exclusion is silent (no warning, no error).

Query A: SELECT * FROM patients WHERE age > 40
Query B: SELECT * FROM patients WHERE age > 40 OR age IS NULL
Query C: SELECT COUNT(*) FROM patients
Query D: SELECT COUNT(age) FROM patients
Query E: SELECT AVG(age) FROM patients

→ Queries C and D are particularly instructive. COUNT(*) counts all rows including NULLs. COUNT(age) counts only non-NULL values. The difference is the number of patients with unknown age. AVG(age) silently excludes NULLs — it computes the average of known ages only, with no indication that some ages were missing.

Key question: A hospital reports "the average age of our patient population is 52." If 20% of age fields are NULL, is this statement accurate? What would be a more honest way to report it?

EX 02
📊 When Mean Imputation Lies
Use Demo 2 to find the distribution shape where mean imputation is most harmful — and explain why.

In Demo 2, switch to the Bimodal distribution. This represents a patient population with two distinct age groups — perhaps pediatric and adult wards combined. Set missing percentage to 30%.

→ Apply mean imputation. The mean lands in the valley between the two peaks — a value that almost nobody in the real population has. Where does the imputed value cluster on the histogram? What does this do to analyses that depend on the age distribution?

→ Switch to median imputation. The median is at roughly the same problematic location. What would be the honest approach for a bimodal distribution? (Hint: what does the analyst need to know before choosing a strategy?)

Key insight: No imputation strategy is universally correct. The right choice depends on the mechanism of missingness (why is the data missing?), the shape of the underlying distribution, and the downstream analysis. The worst outcome is choosing a strategy without thinking about any of these — which is what most automated pipelines do.

EX 03
🔍 The Deduplication Dilemma
Use Demo 3 to find the threshold where the cost of false positives equals the cost of false negatives for a medical context.

In Demo 3, start at threshold 50% and slowly raise it to 95%. Track how precision and recall change. At what threshold does precision first reach 100%? What is recall at that point?

→ In a patient record system, a false positive merge (merging two different patients) is catastrophic — it could mean a patient receives another person's medication history or test results. A false negative (failing to merge the same patient's duplicate records) means fragmented care — the doctor doesn't see the full history.

→ Given these stakes, should the threshold be higher or lower than you'd use for, say, deduplicating a mailing list? Write a one-sentence policy: "For patient record deduplication, the threshold should be ___ because ___."

Bridge to Lesson 5: This is the same precision/recall tradeoff you saw in the confusion matrix lesson. The domain determines which error is more costly — and that determines the operating point. The math is the same; the stakes are different.

05

Discussion Questions

  • The Mechanism of Missingness. Statisticians distinguish three types of missing data: MCAR (missing completely at random), MAR (missing at random — correlated with observed variables), and MNAR (missing not at random — correlated with the missing value itself). An income field that is more likely to be blank for high earners is MNAR. Why does the mechanism of missingness matter for choosing an imputation strategy? Give an example where ignoring the mechanism produces a biased result.
  • Codd's NULL Debate. Chris Date, one of the most prominent relational database theorists, argued that Codd's NULL was a mistake and should be abolished — replaced by two separate markers: one for "inapplicable" and one for "unknown." Date pointed out that three-valued logic creates logical paradoxes and inconsistencies in SQL. Do you think a single NULL was the right design decision? What would a database with two distinct absence markers look like in practice?
  • Cleaning as Interpretation. Every data cleaning decision is an interpretive act — a claim about what the "true" value is or should be. When you impute a missing age with the median, you are asserting that this patient is probably around the median age. When you flag a $999,999 salary as an entry error and correct it to $99,999, you are asserting the extra 9 was a typo. These assertions could be wrong. Who is responsible when a downstream model makes a harmful decision based on data that was cleaned incorrectly?
  • The Reproducibility Crisis. A 2016 study found that over 70% of researchers had failed to reproduce another scientist's results, and over 50% had failed to reproduce their own. Data cleaning choices — which NULLs to drop, which outliers to cap, which duplicates to merge — are rarely documented in published papers, making exact reproduction impossible. What should the standard for data cleaning documentation be in scientific publications? How does this connect to the broader "garbage in" problem?
  • Connection Forward. In Era 3 you will study statistical learning — training models on data rather than encoding rules. Every technique you will learn assumes the training data is a faithful sample of the real-world distribution. If the data has been cleaned in ways that introduce systematic bias — NULLs dropped non-randomly, outliers capped, duplicates merged incorrectly — the model learns from a corrupted sample. Can you give a specific example of a cleaning decision from this lesson that would produce a biased model if applied carelessly?

Terminology

NULLCodd, 1970
A marker in a relational database representing the absence of a value — distinct from zero, empty string, or false. Can mean not collected, not applicable, or unknown. These three meanings look identical in the database but imply different cleaning strategies. NULL propagates through arithmetic: NULL + 5 = NULL.
Three-Valued LogicTRUE / FALSE / NULL
SQL's logical system in which expressions can evaluate to TRUE, FALSE, or NULL (unknown). WHERE clauses only pass rows where the condition is TRUE — NULL evaluates as "not TRUE" and the row is silently excluded. NULL = NULL evaluates to NULL, not TRUE. Use IS NULL and IS NOT NULL to test for the absence of a value.
WHERE age != 30 -- silently drops NULL ages WHERE age != 30 OR age IS NULL -- includes unknowns
ImputationFilling missing values
The process of replacing NULL values with estimated values. Common strategies: mean imputation (replace with column average — harmful for skewed or bimodal distributions), median imputation (more robust to outliers), mode imputation (for categorical data), model-based imputation (predict the missing value from other columns). The right choice depends on the mechanism of missingness and the downstream analysis.
Entity ResolutionRecord linkage / deduplication
The problem of determining when two records in a dataset refer to the same real-world entity. Exact matches (same ID, same name) are easy. Fuzzy matching (similar names, same address) requires a similarity metric (edit distance, token overlap, phonetic matching) and a threshold above which records are merged. The threshold controls the precision/recall tradeoff — lower threshold → higher recall, more false merges.
Edit DistanceLevenshtein distance
The minimum number of single-character edits (insertions, deletions, substitutions) needed to transform one string into another. "Smith" to "Smyth" = 1 edit (i→y). "John" to "Jon" = 1 deletion. Used in fuzzy deduplication to measure name similarity. Normalized edit distance = edit distance / max(len(s1), len(s2)), giving a similarity score in [0, 1].
MCAR / MAR / MNARMissing data mechanisms
Missing Completely At Random (MCAR): missingness is unrelated to any variable — safe to drop or impute. Missing At Random (MAR): missingness depends on observed variables — imputation conditioned on those variables is valid. Missing Not At Random (MNAR): missingness depends on the missing value itself — no imputation strategy is unbiased without additional assumptions. The mechanism must be understood before choosing a cleaning strategy.
